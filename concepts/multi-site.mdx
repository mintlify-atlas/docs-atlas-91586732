---
title: "Multi-site architecture in Camaleon CMS"
description: "Run multiple independent sites from one Camaleon CMS install. Each site has its own domain, theme, plugins, users, and content sharing a single database."
---

Camaleon CMS supports running multiple independent sites from a single Rails application and a single database. Each site has its own post types, content, categories, tags, nav menus, widgets, custom field groups, media library, theme, and plugin set. Sites are distinguished by domain name or subdomain. All isolation happens at the model level — there are no separate databases or Rails processes required.

## How site routing works

On every web request, the `current_site` helper (defined in `CamaleonCms::SiteHelper`) identifies the active site:

1. If the global `$current_site` is set (console or background job use), that site is returned immediately.
2. If `@current_site` is already set on the controller instance, it is returned.
3. If only one site exists in the database, that site is used without any domain matching.
4. For multi-site installations, the request hostname (parsed domain + subdomain) is matched against the `slug` column of the `camaleon_cms_term_taxonomies` table where `taxonomy = 'site'`.

```ruby
# From CamaleonCms::SiteHelper#current_site
host = [request.original_url.to_s.parse_domain]
host << request.subdomain if request.subdomain.present?
site = CamaleonCms::Site.where(slug: host).first.decorate
```

<Note>
  The `slug` field on a site model is its domain identifier. For the main (first) site, `slug` is the bare domain (e.g. `example.com`). For additional sites, `slug` can be a subdomain (e.g. `shop`) — in which case the full domain resolves to `shop.example.com` — or a fully qualified domain (e.g. `shop.example.com` or `other-domain.com`).
</Note>

## Per-site isolation

Each `Site` record owns:

| Resource | Association |
|---|---|
| Post types | `has_many :post_types` |
| Posts | `has_many :posts, through: :post_types` |
| Nav menus | `has_many :nav_menus` |
| Widgets and sidebars | `has_many :widgets`, `has_many :sidebars` |
| Plugins | `has_many :plugins` |
| Themes | `has_many :themes` |
| Custom field groups | `has_many :custom_field_groups` |
| Media | `has_many :public_media`, `has_many :private_media` |
| User roles | `has_many :user_roles_rel` |

## User sharing between sites

The `users_share_sites` key in `config/system.json` controls whether users are shared across all sites or scoped to a single site.

```json
{
  "users_share_sites": true
}
```

| Value | Behavior |
|---|---|
| `true` (default) | All users are visible on every site. User roles are read from the main site's roles. |
| `false` | Each user belongs to one site. Users and their roles are isolated per site. Destroying a site also destroys its users. |

When `users_share_sites` is `true`, `site.users` returns `CamaleonCms::User.all`. When it is `false`, it returns only users with `site_id` matching the current site.

## Adding a new site from the admin panel

<Steps>
  <Step title="Log in as a super-admin">
    Only super-admins can manage sites. Navigate to the admin panel of the main site.
  </Step>
  <Step title="Open site management">
    Go to **Settings → Sites** and click **New Site**.
  </Step>
  <Step title="Set the slug / domain">
    Enter the domain or subdomain slug. For a subdomain site on `example.com`, enter `shop` (the full URL will be `shop.example.com`). For an independent domain, enter the full domain such as `shop.example.com`.
  </Step>
  <Step title="Choose a theme and plugins">
    After the site is created, switch to its admin context and install a theme and any required plugins under **Appearance** and **Plugins**.
  </Step>
</Steps>

## Switching the active theme per site

Each site independently tracks its active theme via the `_theme` option:

```ruby
# Read the current theme slug
current_site.get_theme_slug  # => "default"

# Switch to a different theme (triggers on_inactive / on_active hooks)
site_install_theme("my_custom_theme")
```

The theme model is looked up from the site's `themes` association:

```ruby
current_site.get_theme           # => decorated Theme for current theme
current_site.get_theme("blog")   # => decorated Theme with slug "blog"
```

## Setting $current_site in console or background jobs

When running code outside of a web request (Rails console, Rake tasks, Sidekiq jobs), there is no HTTP request to derive the site from. Set the `$current_site` global before calling any CMS methods:

```ruby
# Rails console or background job
$current_site = CamaleonCms::Site.first.decorate

# Now current_site is resolved correctly
current_site.post_types.each { |pt| puts pt.name }
```

To target a specific site by domain slug:

```ruby
$current_site = CamaleonCms::Site.find_by(slug: "shop.example.com").decorate
```

Reset after you are done if you are in a long-running process that handles multiple sites:

```ruby
$current_site = nil
```

## Useful Site model methods

```ruby
site = CamaleonCms::Site.find_by(slug: "example.com")

site.main_site?         # => true if this is the first (main) site
site.get_domain         # => "example.com" or "shop.example.com"
site.get_theme_slug     # => "default"
site.front_per_page     # => 10 (items per page on frontend)
site.admin_per_page     # => 10 (items per page in admin)
site.is_active?         # => true
site.is_maintenance?    # => false
site.get_languages      # => [:en, :es]

# Return the main site (first site by ID)
CamaleonCms::Site.main_site
```
